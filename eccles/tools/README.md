# Eccles TTS Packager

> ⚠️ **Debug build in progress.** Only the PlatformIO-driven build path is currently functional —
> see [Known Issues](#known-issues) before relying on manual command-line usage.

## What it does

The packager scans a folder of `.wav` clips, optionally re-encodes each one (sample rate, bit
depth, channel count, trailing-silence removal) according to a small text config file, concatenates
the results into a single binary blob, and generates a C++ header so the ESP32 firmware can look
up any clip by name at runtime with zero heap allocation.

Two files come out of a successful run:

1. **`StaticModel.eccles`** — the binary clip bundle.
2. **`StaticModel.h`** — a generated, do-not-edit C++ header indexing that bundle.

### Binary bundle layout

```
["ECCLES" 6-byte magic][clip 1 audio payload][clip 2 audio payload]...[clip N audio payload]
```

There's no per-clip header inside the binary itself — clip boundaries (name, byte offset, byte
size) live entirely in the generated `StaticModel.h`, not in the `.eccles` file.

### Generated header

```cpp
// Auto-generated by eccles-tts-packager. DO NOT EDIT.
ECCLES_API {
    static const e_uint32 ECCLES_SAMPLE_RATE = 8000;
    static const e_uint16 ECCLES_BIT_DEPTH   = 16;
    static const e_uint16 ECCLES_CHANNELS    = 1;

    typedef struct { e_string name; e_uint32 offset; e_uint32 size; } model_t;

    model_t HELLO = { .name = "hello", .offset = 6U, .size = 4096U };
    // ... one instance per clip ...

    model_t fromText(e_string text){
        switch(eccles_hashRT(text)){
            case 2219287063U: return HELLO; // hello
            // ...
            default: return {nullptr,0,0};
        }
    }
};
```

Notable design choice: clip-name hashes (`djb2`) are computed **once, at pack time**, by the
packager's own copy of the hash function, and written into the generated header as plain integer
case labels (`case 2219287063U:`). The firmware never has to call a hash function for this table
at all — it's a free jump table the compiler builds from literal constants. The firmware's own
`eccles_hashRT()` (in `EcclesTypes.h`) just has to compute the same algorithm at runtime for a
match to land on the right case.

Call `fromText("hello")` (no extension) from firmware code to get the matching `model_t`.

## Config file — `EcclesConfig.txt`

Place this next to the packager binary (or point `-config.txt` at it manually). **None exists in
this repo yet** — until you add one, every build uses the defaults below.

| Key | Values | Default | Meaning |
|---|---|---|---|
| `raw` | `0` / `1` | `0` | If `1`, skip all reprocessing — pack the `.wav` payload byte-for-byte as-is. |
| `rate` | integer Hz | `8000` | Target sample rate. Triggers linear-interpolation resampling if it differs from the source. |
| `depth` | `8` / `16` / `24` / `32` | `16` | Target bit depth. Converting depth applies TPDF dithering. |
| `channel` | `mono` / `stereo` | `mono` | Target channel count. |
| `tsr` | `true` / `false` | `true` | Trim trailing silence from the end of each clip. |
| `type` | `static` / `dynamic` | `static` | **Only `static` is currently implemented** — see Known Issues. |

Example:
```
rate:8000
depth:16
channel:mono
tsr:true
```

## Building & running

### Automatic (recommended) — via PlatformIO

The `[env:native]` PlatformIO environment builds this packager for your host machine, then
`execute-tts-packager.py` (a post-build script) takes over:

1. Hashes `eccles/resources/models/` — if nothing changed since last time, it skips the rest.
2. Copies the freshly built executable to `eccles/tools/bin/eccles-tts-packager.exe`.
3. Runs it with the `-platformIO` flag from inside `eccles/tools/bin/` (paths inside the packager
   are resolved relative to that working directory).
4. On success, uploads the regenerated filesystem image to the ESP32 (`esp32dev` env, `uploadfs`
   target) and records the new models-folder hash so the next build can skip unchanged work.

```bash
pio run -e native
```

### Manual / standalone

The packager's argument parser supports:

```
eccles-tts-packager -type [static|dynamic] <folder> [config.txt]
```

— but **this currently does nothing**. See Known Issues #1.

## Known Issues

1. **Manual CLI invocation is non-functional.** `eccles_main()` parses `-type`, the folder, and
   any `.txt` config path into `globalState`, but the dispatch at the end of `main.cpp` is:
   ```cpp
   if(globalState.platformIO){
       staticModel sm;
       return sm.pack() ? 0 : 1;
   }
   return 1;
   ```
   Without `-platformIO`, the parsed arguments are never used — the program just returns failure.
   Standalone packaging from a terminal isn't wired up yet despite being parsed.
2. **`staticModel::extract()` is a stub** — always returns `false`. Unpacking an existing
   `.eccles` bundle back into individual `.wav` files isn't implemented.
3. **Dynamic model packing is parsed but unused.** `-type dynamic` sets a local `dynamic` flag in
   `main.cpp` (currently marked `[[maybe_unused]]`), but `staticModel::pack()` only ever produces
   the static bundle format described above — there's no dynamic-format writer wired in yet.
4. **32-bit depth is accepted by `initializeAudioPins()` on the firmware side but largely
   unexercised here** — most testing has been done at 8/16-bit; treat 24/32-bit conversion as
   unverified until someone runs it through a full pack → flash → playback cycle.
5. This document replaces the previous plain-text `ReadME`, which referenced an `ffmpeg.exe`
   dependency and a `config.eccles` filename that don't match the current implementation (the
   packager does its own resampling/depth conversion in `AudioConfig`/`Resampler`, and the config
   file is `EcclesConfig.txt`).
